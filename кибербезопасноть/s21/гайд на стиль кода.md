# Руководство по разработке на C в Школе 21

## 1. Принципы структурного программирования Э. Дейкстры

Эти правила проверяются как линтерами, так и студентами во время P2P-оценок.

### 📏 Ограничение на размер функций (Максимум 42 строки)
* **Правило**: Любая функция должна занимать **не более 42 строк** (от открывающей до закрывающей фигурной скобки).
* **Зачем**: Это заставляет разбивать сложную логику на мелкие независимые подпрограммы (декомпозиция).
* **Как делать**: Если ваша функция разрослась, выносите логические блоки в отдельные вспомогательные функции.

### 🚫 Запрет на глобальные переменные
* **Правило**: Переменные нельзя объявлять вне функций. Использование ключевых слов `static` (для глобальных) также запрещено.
* **Исключение**: Глобальные константы (`#define WIDTH 80`) разрешены.

### ↩️ Единственная точка выхода (One Exit Rule)
* **Правило**: В функции должен быть только **один оператор `return`** (в самом конце).
* **Исключение**: Предварительная проверка аргументов в самом начале функции. Но и её по возможности рекомендуется избегать.
* **Пример**:
  ```c
  // ❌ ПЛОХО (несколько return)
  int min(int a, int b) {
      if (a < b) return a;
      return b;
  }

  //  ХОРОШО (один return в конце)
  int min(int a, int b) {
      int result = b;
      if (a < b) {
          result = a;
      }
      return result;
  }
  ```

### 🔁 Правила для циклов
* **Один выход из цикла**: Цикл должен завершаться только по условию в его заголовке (например, `while (condition)`).
* **Запрет на `continue`**: Использование `continue` строго запрещено (так как по поведению напоминает `goto`).
* **Запрет на `break`**: `break` можно использовать только в бесконечных циклах в качестве единственной точки выхода. Во всех остальных случаях управляйте циклом через переменные в условии.

### 📦 Ограничение вложенности (Максимум 4 уровня)
* **Правило**: Вложенность блоков кода (`if`, `while`, `for`) внутри функции не должна превышать **4 уровней**. Отсчет начинается изнутри тела функции.
* **Как делать**: Если уровень вложенности приближается к критическому, выносите самый глубокий цикл или проверку во вспомогательную функцию.

### ❌ Запрет на `exit()`
* **Правило**: Программа должна завершаться естественным выходом из `main` через `return 0`. Принудительное завершение через `exit()` запрещено.

## 2. Форматирование кода через `clang-format`

Для стилевых тестов используется утилита `clang-format`.

> [!IMPORTANT]
> Файл конфигурации `.clang-format` обязательно должен лежать в директории `src/` вашего проекта. Тогда правила форматирования будут автоматически применяться к исходным файлам.

### Полезные команды в терминале:

1. **Проверка стиля (без изменения файла)**:
   ```bash
   clang-format -n src/pong.c
   ```
   *Если команда ничего не вывела — код отформатирован идеально. Если вывела предупреждения — их нужно исправить.*

2. **Автоматическое форматирование файла**:
   ```bash
   clang-format -i src/pong.c
   ```
   *Эта команда автоматически расставит правильные отступы, переносы строк и пробелы в соответствии с правилами Google Style.*

## 3. Компиляция и сборка кода

Автотестер всегда собирает код с очень строгими флагами. Любое предупреждение (warning) компилятора прерывает сборку.

### Команда для компиляции:
```bash
gcc -Wall -Wextra -Werror src/pong.c -o pong
```

### Значение флагов:
* `-Wall`: Включает основные предупреждения компилятора о потенциальных ошибках.
* `-Wextra`: Включает дополнительные предупреждения (например, неиспользуемые параметры функций).
* `-Werror`: **Критический флаг**. Превращает любые предупреждения в ошибки сборки (`Build fail`).

> [!TIP]
> При объявлении функций следите, чтобы все аргументы использовались в коде. Если вы передали аргумент, но не воспользовались им, флаг `-Wextra` выдаст ошибку `unused parameter`.

## 4. На что еще важно обратить внимание

### 🔌 Работа со стандартным вводом и EOF (End of File)
При автоматическом тестировании ввод подается в программу из файла или пайпа.
* Если тест закончит подавать команды, поток ввода закроется. 
* Если ваша программа не проверяет возврат функций `scanf` или `getchar` на равенство `EOF` (или `-1`), она уйдет в **бесконечный цикл**, потребляя 100% процессора. Автотестер завершит её по таймауту и выдаст ошибку.
* **Пример правильного считывания**:
  ```c
  int c = getchar();
  if (c == EOF) {
      // Корректно завершаем программу или выходим из игрового цикла
  }
  ```

### 🧹 Очистка буфера ввода
При пошаговом вводе (`scanf` или `getchar`) в буфере часто остается символ переноса строки `\n`.
* Если не очищать буфер, следующий вызов ввода считает этот `\n` как действие игрока, что приведет к неверному поведению.
* **Пример очистки буфера до конца строки**:
  ```c
  int temp;
  while ((temp = getchar()) != '\n' && temp != EOF);
  ```

### 🔤 Соответствие условию задачи
* Если в условии сказано: "вывести поздравление победителя и завершиться", выводите текст ровно в том формате, в каком это требуется.
* Не добавляйте лишних надписей вроде `"Введите ход: "` или `"Ошибка!"`, если они не прописаны в задании или тестах. Иногда автотесты сравнивают вывод посимвольно, и любой неожиданный символ сломает проверку.
